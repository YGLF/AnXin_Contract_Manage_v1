## 1.3 业务流程介绍

系统主要业务流程围绕合同的全生命周期管理展开，包含以下核心环节：

### 1.3.1 合同创建流程

1. 用户登录系统
2. 创建新合同，填写合同基本信息（标题、编号、金额、客户等）
3. 合同状态为"草稿"（draft）
4. 可上传合同相关文档

### 1.3.2 审批流程

1. 用户提交合同审批
2. 合同状态变为"待审批"（pending）
3. 进入三级审批流程：
   - 第一级：销售总监审批
   - 第二级：技术总监审批
   - 第三级：财务总监审批
4. 审批通过后，合同状态变为"已批准"（approved）
5. 合同生效后，状态变为"已生效"（active）

### 1.3.3 合同执行流程

1. 合同执行中状态（in_progress）
2. 记录执行进度和付款记录
3. 执行完成后可发起待付款（pending_pay）
4. 全部完成后状态变为"已完成"（completed）
5. 可申请归档（archived）

### 1.3.4 状态变更审批

对于关键状态变更（归档、终止、执行中、待付款），需要管理员审批通过后才能生效。

# 第二章 密码应用

安信合同管理系统在设计和实现过程中，严格遵循密码应用相关法规和标准，采用多种密码技术保障系统安全。本章节详细介绍系统中所使用的密码技术及其应用场景。

## 2.1 用户认证与访问控制

### 2.1.1 JWT令牌认证

系统采用JSON Web Token（JWT）进行用户身份认证。当用户成功登录后，后端会生成一个包含用户身份信息的JWT令牌，并返回给前端。前端在后续的每次请求中都必须在HTTP请求头中携带该令牌，后端验证令牌有效性后才允许访问受保护的资源。

**JWT认证流程：**

```
┌─────────┐                          ┌─────────┐                         ┌─────────┐
│  用户   │                          │  前端   │                         │  后端   │
└────┬────┘                          └────┬────┘                         └────┬────┘
     │                                    │                                   │
     │  1. 输入用户名密码                 │                                   │
     │──────────────────────────────────▶│                                   │
     │                                    │  2. 发送登录请求                  │
     │                                    │─────────────────────────────────▶│
     │                                    │                                   │
     │                                    │  3. 验证用户名密码                │
     │                                    │  4. 生成JWT Token                │
     │                                    │◀─────────────────────────────────│
     │                                    │                                   │
     │  5. 登录成功，显示Token            │                                   │
     │◀──────────────────────────────────│                                   │
     │                                    │                                   │
     │                                    │  6. 携带Token发送业务请求         │
     │                                    │─────────────────────────────────▶│
     │                                    │                                   │
     │                                    │  7. 验证Token有效性              │
     │                                    │  8. 处理业务请求                  │
     │                                    │◀─────────────────────────────────│
```

**JWT令牌结构：**

JWT令牌由三部分组成，各部分之间用"."分隔：

- **Header（头部）**：包含令牌类型和所使用的签名算法
- **Payload（负载）**：包含用户ID、用户名、角色、权限等信息
- **Signature（签名）**：对前两部分的签名，防止数据被篡改

### 2.1.2 密码存储加密

系统使用bcrypt算法对用户密码进行加密存储。bcrypt是一种基于Blowfish加密算法的密码哈希函数，具有盐值（salt）生成和成本因子（cost factor）设置功能，能够有效抵抗彩虹表攻击和暴力破解。

**bcrypt加密特点：**

- 每次加密会自动生成随机盐值
- 支持设置计算成本（默认cost=10）
- 加密结果不可逆，无法从密文还原明文
- 验证时只需比较加密后的值

### 2.1.3 登录超时机制

系统实现30分钟无操作自动退出机制。用户在登录后，如果30分钟内没有任何操作，Token会自动过期，需要重新登录。同时，系统会在Token即将过期前2分钟弹出提醒，提示用户进行操作以刷新Token有效期。

## 2.2 密码杂凑验证

### 2.2.1 SHA-256密码杂凑

系统在用户登录时使用SHA-256算法对密码进行杂凑处理。前端在用户输入密码后，首先使用SHA-256对密码进行杂凑，然后将杂凑值发送到后端进行验证。这种方式可以防止密码在网络传输过程中被明文截获。

**SHA-256杂凑流程：**

```
明文密码 ──▶ SHA-256杂凑 ──▶ 杂凑值 ──▶ 传输到后端验证
```

**SHA-256特点：**

- 固定输出长度256位
- 单向函数，不可逆
- 碰撞抵抗性强
- 适用于数据完整性验证

### 2.2.2 数据完整性校验

系统对用户鉴别信息使用SHA-256生成完整性哈希值，用于检测数据在传输或存储过程中是否被篡改。每次读取用户数据时，都会重新计算哈希值并与存储的哈希值进行比对，确保数据完整性。

## 2.3 数据存储加密

### 2.3.1 加密服务架构

系统提供灵活的数据加密服务接口，支持多种加密算法，包括：

- **SM4**：国密对称加密算法
- **AES**：高级加密标准（国际算法）
- **HSM**：硬件安全模块/服务器密码机（预留接口）

### 2.3.2 SM4国密加密

SM4是中国国家密码管理局发布的对称加密算法，为无线局域网产品提供加密支持和国产密码算法支持。SM4算法支持128位密钥长度，具有较高的安全强度。

**SM4加密API：**

```
POST /api/crypto/encrypt
Content-Type: application/json
Authorization: Bearer <token>

{
    "data": "待加密的敏感数据",
    "algo_type": "sm4"
}

响应：
{
    "cipher_text": "加密后的密文",
    "algorithm": "sm4"
}
```

### 2.3.3 AES国际加密

AES（Advanced Encryption Standard）是美国联邦政府采用的一种对称加密标准。AES加密速度快，安全性高，广泛应用于各种安全场景。

**AES加密API：**

```
POST /api/crypto/encrypt
Content-Type: application/json
Authorization: Bearer <token>

{
    "data": "待加密的敏感数据",
    "algo_type": "aes"
}
```

### 2.3.4 加密服务管理

系统提供统一的加密服务管理接口，支持动态配置加密算法和密钥：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/crypto/config-sm4` | POST | 配置SM4加密密钥 |
| `/api/crypto/config-aes` | POST | 配置AES加密密钥 |
| `/api/crypto/config-hsm` | POST | 配置HSM密码机（预留） |
| `/api/crypto/status` | GET | 查看加密服务状态 |

## 2.4 权限控制

### 2.4.1 用户级别细粒度权限

系统采用"角色权限+用户自定义权限"的双重权限控制模式：

- **角色权限**：定义每个角色默认拥有的权限
- **用户自定义权限**：基于角色权限追加额外权限（追加模式）
- **最终权限** = 角色权限 ∪ 用户自定义权限

### 2.4.2 系统角色

| 角色 | 标识 | 默认权限 |
|------|------|----------|
| 超级管理员 | admin | 所有权限 |
| 经理 | manager | 仪表盘、合同权限、客户权限、审批权限 |
| 销售 | user | 仪表盘、查看/创建合同、查看/创建客户 |
| 审计管理员 | audit_admin | 仪表盘、查看审计、查看合同/客户/审批 |

### 2.4.3 权限验证机制

系统实现了路由守卫和API中间件双重权限验证：

1. 前端路由守卫根据路由配置的权限检查用户是否有权访问
2. 侧边栏菜单根据用户权限动态显示/隐藏菜单项
3. 后端API中间件可添加权限验证，确保API访问安全

## 实训四：密码杂凑验证

### 4.1 实训目的

1. 理解SHA-256密码杂凑算法原理
2. 掌握前端密码杂凑处理流程
3. 能够使用工具验证密码杂凑结果

### 4.2 实训原理

SHA-256（Secure Hash Algorithm 256-bit）是一种密码杂凑函数，生成256位（32字节）的杂凑值。

**特点：**

- 单向函数：无法从杂凑值还原原始数据
- 固定输出：任何长度的输入都产生256位输出
- 碰撞抵抗：很难找到两个不同的输入产生相同的杂凑值

### 4.3 实训步骤

#### 步骤1：分析前端代码

1. 找到前端登录代码中密码处理部分
2. 路径：`frontend/src/views/Login.vue`
3. 找到密码杂凑相关代码：

```javascript
// 使用SHA-256对密码进行杂凑
const hashPassword = (password) => {
    return CryptoJS.SHA256(password).toString();
}
```

#### 步骤2：手动计算杂凑值

1. 打开在线SHA-256工具或使用命令行
2. 输入密码：admin@123456
3. 获取SHA-256杂凑值

命令行方式：

```bash
echo -n "admin@123456" | sha256sum
```

#### 步骤3：验证登录流程

1. 打开浏览器开发者工具
2. 在登录页面输入用户名和密码
3. 点击登录
4. 在Network中查看请求内容
5. 确认密码字段已转换为杂凑值传输

#### 步骤4：数据完整性校验

1. 登录系统后，进入用户管理
2. 查看用户信息
3. 尝试修改用户数据
4. 观察系统如何验证数据完整性

### 4.4 实训结果

完成本实训后，应能够：

- 理解SHA-256算法原理
- 掌握密码杂凑在认证中的应用
- 能够验证密码杂凑结果

## 实训五：数据存储加密服务

### 5.1 实训目的

1. 掌握SM4国密对称加密算法使用
2. 掌握AES国际对称加密算法使用
3. 理解加密服务API的调用方式

### 5.2 实训原理

对称加密算法使用相同的密钥进行加密和解密：

- **SM4**：国密算法，128位密钥
- **AES**：国际算法，256位密钥（常用）

**加密流程：**

```
明文数据 ──▶ 加密算法(Key) ──▶ 密文数据
```

**解密流程：**

```
密文数据 ──▶ 解密算法(Key) ──▶ 明文数据
```

### 5.3 实训步骤

#### 步骤1：配置SM4加密密钥

使用管理员账号，配置SM4加密密钥：

```bash
curl -X POST http://localhost:8000/api/crypto/config-sm4 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <admin_token>" \
  -d '{"key": "1234567890123456"}'
```

响应：

```json
{"message": "SM4 configured successfully"}
```

#### 步骤2：使用SM4加密数据

```bash
curl -X POST http://localhost:8000/api/crypto/encrypt \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{"data": "这是敏感合同数据", "algo_type": "sm4"}'
```

响应示例：

```json
{
    "cipher_text": "U2FsdGVkX1+...",
    "algorithm": "sm4"
}
```

#### 步骤3：使用SM4解密数据

```bash
curl -X POST http://localhost:8000/api/crypto/decrypt \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d '{"data": "U2FsdGVkX1+...", "algo_type": "sm4"}'
```

响应：

```json
{"plain_text": "这是敏感合同数据"}
```

#### 步骤4：配置AES加密

1. 配置AES密钥（32位）：

```bash
curl -X POST http://localhost:8000/api/crypto/config-aes \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <admin_token>" \
  -d '{"key": "12345678901234567890123456789012"}'
```

2. 使用AES加密数据（algo_type设为"aes"）

#### 步骤5：查看加密服务状态

```bash
curl http://localhost:8000/api/crypto/status \
  -H "Authorization: Bearer <token>"
```

响应：

```json
{
    "hsm_configured": false,
    "sm4_configured": true,
    "aes_configured": true
}
```

### 5.4 实训结果

完成本实训后，应能够：

- 描述对称加密的基本原理
- 掌握SM4和AES加密API的使用方法
- 理解加密服务在数据存储中的应用
